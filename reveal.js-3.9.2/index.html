Puertas lógicas

3 tipos de puertas lógicas:

Exclusivo
Paralelo
Inclusivo

  • Las condiciones se definen en las transiciones salientes. No hay condiciones para puertas paralelas
  • Una buena práctica es usar las puertas simétricamente para mejorar la legibilidad
5

Exclusivo: usa la transición predeterminada y se asegúra de que solo una condición sea verdadera entre las transiciones salientes
Exclusivo: las nuevas transiciones no tienen condición por defecto, por lo que obtienes un error de validación en la puerta de enlace. Hay que definir la condición y activar la validación para deshacerse del error.
Exclusivo: solo una transición entrante debe llegar a la puerta de enlace.
Paralelo: todas las transiciones salientes se disparan al mismo tiempo (no se permite ninguna condición).
Paralelo: se esperan todas las transiciones entrantes, incluso si alguna nunca se puede alcanzar debido a un mal diseño.
Inclusivo: se utiliza cuando se pueden realizar varias transiciones según las condiciones
Mejor práctica: explícita, simétrica y predeterminada
Consulta la especificación BPMN, figura 13.1, página 427 (457 en PDF)

  • Específicos para el proceso

Modelar los datos: el modelo de datos de negocio (BDM)

  • Almacena datos de negocio relacionados con procesos y/o aplicaciones
  • El modelo definido de objetos para almacenar datos de negocio
  • El modelo se define en el estudio Bonita
  • Desde el modelo, Bonita genera clases Java (POJO) y crea las tablas en la base de datos
Complex type with fields. Structured objects composed of: Java classes (Integer, Boolean, String, Date...) and other BDMs
Java POJO generated out of the model.
Stored in a dedicated database with automatic administration (tables creation, updates...).
Persistence handle by Bonita using Hibernate library and the database.
Transactional database storage guarantees that data is safely kept in case of server shut down (expected or not).

El modelo de datos de negocio(BDM): uso

  • Declarar variables de negocio en la definición del proceso
  • Crear (Create) / Leer (Read) / Actualizar (Update) / Eliminar (Delete) usando: valor predeterminado, operaciones, conectores ...
  • Se puede compartir entre diferentes instancias de proceso
  • Ciclo de vida controlado por el desarrollador
Business variable uses BDM definition to define the type of the variable.
Several solutions exist to Create (INSERT) / Read (SELECT) / Update (UPDATE) / Delete (DELETE).
Business variables lifecycle is independent of the lifecycle of process instance. The data can be updated by several processes.
Accessible by REST APIs if integration is needed with third party application or for application pages.
Never try to read or write data directly from the BDM database, use the APIs to do so.

Datos: del proceso a la interfaz de usuario

El ciclo de datos desde la inicialización hasta la actualización

2.30

When the user inputs some data in a form they are first stored in JavaScript variables.
When user click on the submit button, a REST API call is performed to the Engine API to instrcut the Engine to start a new process instance or execute a task. This call carry the value of a JavaScript variable associated with the submit button. REST API call is an HTTP request that carry out data as JSON.
Engine will check that received JSON data match the contract definition and validation rules. If it match, the data are now stored in the contract variables.
Contract variables can then be used in various place such as a business variable default value (this will create a new row in business database), operation on a task to change the attribute value of a businnes variable (this will perform an UPDATE SQL request on the business database)...

Los contratos

Conjunto de parámetros de entrada requeridos y reglas de validación

Los datos enviados por los formularios deben cumplir con los contratos para:

  • Instanciar un proceso
  • Realizar una tarea humana

Contribuye al desacoplamiento entre la lógica empresarial y las interfaces gráficas

Actores en el proceso

Para definir quién puede hacer una tarea

Estos actores se asociarán entonces a la organización definida según:

  • Grupos
  • Roles
  • Usuarios

Actores

  • Declarar uno o más actores en la definición del proceso (solo un nombre)
  • Configurar para asociar al actor con los usuarios:
    • Directamente a un conjunto determinado de usuarios
    • Indirectamente a través de grupos y roles
  • Asociar a los actores directamente con la tarea o a través de un carril (lane)
  • Los actores no pueden usar datos "en vivo"
Declare actor by selecting the process pool.
Actor can be mapped to a subset of the organization using actor mapping: lists of users, groups, roles and memberships
Organization changes are immediately reflected on who can do a task. Actors allow to automatically update task visibility when organization changes.
[Studio & Portal - edit process configuration actor mappings, run it]
Actors cannot use any live data (business variables, external system information...). We can do that with filters

¿Quién puede iniciar una instancia de proceso?

All users associated with the actor with "initiator" flag

Only one actor on a given process definition has this flag

For a given process instance the user who start it will be registered as the initiator of the instance

A specific actor is related to process instantiation
