Bonita BPM Engine

Encargado de la ejecución de los procesos

Basado en tecnologias relacionadas con Java 7 :

  • JDBC (via Hibernate y data sources)
  • Java EE 6 : JSP, Servlet, JTA ...
  • Spring
  • Quartz

Empaquetado junto al Bonita Portal en una aplicación web Java

Necesita :

  • Un servidor de aplicaciones Java
  • Una Base de datos transaccional JDBC
No dispone de UI
Todas las interacciones con el Engine desde otros componentes (Portal, otros) se gestionan con las Bonita APIs.

Bonita BPM Portal

Interfaz web para el usuario final y administradores

Muestra la lista de tareas disponibles

Permite desplegar y configurar nuevos procesos

Usuarios: Iniciar un proceso, ejecutar tareas
Administradores: desplegar y configurar procesos, gestionar permisos, monitorizar la ejecucion de los procesos
El Portal usa las Bonita APIs como cliente del Engine

Introducción al BPMN

Notación estandard (OMG) para el modelado de procesos

Un proceso se representa por un Pool

Elementos principales:

  • Evento Inicio
  • Tarea Humana
  • Tarea Automática
  • Evento de Fin
OMG = Object Management Group, creo diversos estandares como UML
Shapes (BPMN standard): circle = event, rectangle = activity
Colors (Bonita specific): green = start, blue = intermediate, red = end

Gestión de Datos : Ámbito

3 niveles de visibilidad para los datos:

  • Proceso
  • Tarea
  • Formulario
- Los datos de un ámbito superior pueden ser accedidos, no al reves.
- Los datos se adjunta a cada instiancia de proceso.
- El ciclo de vidad de los datos esta relacionado con el ambioto, una vez se abandona el ambito, los datos no pueden modificarse
- Si se quieren compartir datos de negocio entre distintos procesos, se debe utilizar una BD externa o un BDM (SP).

Gestión de Datos : Persistencia y Tipos

Todos los datos están almacenados en Base de Datos por el Engine excepto los datos transitorios de tarea o de formulario

Se puede utilizar cualquier tipo Java Serializable

Los tipos primitivos o Strings se serializan para persisitirlos

DB storage guarantees that data is safely kept in case of server shut down (expected or not).
Never try to R/W data directly from Bonita DB, use the APIs to do so.

Introducción BPMN (Continuación)

3 tipos de compuertas :

  • Exclusiva
  • Paralela
  • Inclusiva

Buenas prácticas: las compuertas deberían ser simetricas cuando sea posible

- Exclusive: use default TR and make sure only one condition is true among outgoing TRs
- Exclusive: only one incoming TR should reach the gateway.
- Parallel: all outgoing TR fired at the same time (no condition allowed).
- Parallel: all incoming TR are waited for even if some can never be reached due to bad design.
- Inclusive: used when dealing with parallel gateways with conditions

Formularios

2 tipos de formularios:

  • Formulario de instanciación y de tarea
  • Formulario de resumen

Un formulario es un conjunto de widgets ubicados en una grilla

Case start and overview form are defined at pool level.
Click on submit button in forms triggers the completion of human tasks or the start of a process.
Overview form is always available: whether the process is running or completed.
Multiple page forms: pageflows

Widgets

Los widgets tienen disponible una serie de opciones de configuración :

  • Nombre (id) y etiqueta
  • Ayuda, tooltip, ejemplo
  • Valores iniciales, disponibles
  • Operación de salida
  • Inserción condicional, obligatorio, solo lectura...
  • Validaciones

Actores

2 opciones de implementación :

  • Actores : mapeo entre la definición del proceso y la organización
  • Filtros de Actor : lógica ejecutada cuando se prepara la tarea
Actors allow to automatically update task visibility when organization changes.
Actors cannot use process data whereas filters can.
Filters are executed only once when task starts.

Conectores

Permite a los procesos interactuar con otros sistemas externos (lo contrario que las APIs que permiten gestionar el Engine)

El Studio dispone de un conjunto de conectores

Se pueden implementar conectores personalizados tanto desde el Studio como de otros Java IDEs

Un conector se compone de:

  • una definición (Java interface) que especifica las entradas/salidas del conector
  • una implementación (Java class) que define la lógica de negocio
Connectors can be placed at the start or the end of a process or a task.
Any data type or groovy script can be used as I/O.
When possible prefer connectors over scripts for increased performance, error handling (SP).
