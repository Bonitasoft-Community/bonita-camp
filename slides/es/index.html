Bonita BPM Studio : Editor de Procesos

El Studio esta montado sobre Eclipse. Requiere JVM 7 trabajando en la compatibilidad con la 8
Permite la creación de diagramas de proceso que pueden contener 1 o varios procesos
Diferentes etapas: modelado del proceso, configuracion de datos, diseño de formularios, configuracion de conectores...

Bonita BPM Studio : UI Designer

Basado en Navegadores Web. Usa la tecnología de AngularJS y Bootsrap. Permite la creación de widgets personalizados.

Bonita BPM Engine

Encargado de la ejecución de los procesos (no built-in UI)

Basado en tecnologias relacionadas con Java 7

Empaquetado junto al Bonita Portal en una aplicación web Java

Necesita :

  • Un servidor de aplicaciones Java
  • Una Base de datos transaccional JDBC
Todas las interacciones con el Engine desde otros componentes (Portal, otros) se gestionan con las Bonita APIs.

Bonita BPM Portal

Interfaz web para el usuario final y administradores

Usuarios: Iniciar un proceso, ejecutar tareas
Administradores: desplegar y configurar procesos, gestionar permisos, monitorizar la ejecucion de los procesos
El Portal usa las Bonita APIs como cliente del Engine

Introducción al BPMN

Notación estandard (OMG) para el modelado de procesos

Un proceso se representa por un Pool

Elementos principales:

Evento Inicio
Tarea Humana
Tarea Automática
Evento de Fin

Los elementos se enlazan por Transiciones →

OMG = Object Management Group, creó diversos estandares como UML
Formas (BPMN standard): círculo = evento, rectángulo = actividad o tarea
Colores (Específicos de Bonita): verde = inicio, azul = intermedio, rojo = fin

Gestión de Datos : Ámbito & Ciclos de vida

BDM (Business Data Model)

Completamente accesibles, almacenados en la BD de Negocio

+ Caso de uso: guarda cualquier tipo de datos manipulados por las instancias de proceso y/o las aplicaciones

Variables de Proceso y de Tarea

Accesibles dentro de un proceso o de una instancia de tarea, almacenadas en la BD del Engine, archivadas con la instancia de proceso

+ Caso de uso: ejecución de conectores en cadena

Variables de Formulario

Accesibles desde un Formulario, nunca persisten

+ Caso de uso: procesos y variables almacenadas mostradas y actualizadas en los formularios

- Los datos de un ámbito superior pueden ser accedidos, no al revés.
- Los datos se adjunta a cada instiancia de proceso.
- El ciclo de vidad de los datos esta relacionado con el ámbito, una vez se abandona el ámbito, los datos no pueden modificarse
- Si se quieren compartir datos de negocio entre distintos procesos, se debe utilizar una BD externa o un BDM (SP).

Gestión de Datos : Tipos

BDM Proceso & Variables de Tareas Variables de formulario
Objetos estructurados compuestos de:
  • Tipos primitivos de Java (String, Integer, Boolean, Date...)
  • Otros BDMs
 Cualquier tipo Java Serializable (Primitivos, Colecciones, tipos personalizados...) Cualquier tipo JavaScript

Tipos especiales: Documentos (archivos) con revisiones pueden ser declarados a nivel de proceso.

El almacenaje en BD garantiza que los datos estarán bien guardados en caso de que el servidor se apague (esperado o no).
Nunca intentes leer o escribir datos directamente a la BD de Bonita, usa las APIs para hacerlo.

Contratos

Conjunto de parámetros de entrada requeridos con reglas de validación.

Los datos enviados por los formularios deben satisfacer el contrato al:

  • iniciar un proceso
  • ejecutar una tarea humana

Contribuye al desacoplamiento de la lógica de proceso y la interfaz de usuario.

BPMN Introducción (Continuación)

3 tipos de compuertas :

Exclusiva
Paralela
Inclusiva

Buenas prácticas: las compuertas deberían ser simétricas cuando sea posible

- Exclusiva: usa una transición por defecto y asegúrate de que sólo una condición es verdadera entre las transiciones salientes.
- Exclusiva: sólo una transición entrante deberían alcanzar la compuerta.
- Paralela: todas las transiciones salientes se disparan al mismo tiempo (no se permiten condiciones).
- Paralela: todas las transiciones entrantes se esperan incluso aunque alguna nunca pueda ser alcanzada debido a un mal diseño.
- Inclusiva: usada cuando tratamos con compuertas paralelas con condiciones

Actores

Definidos por sendas (lanes) o tareas humanas en el diagrama.

2 opciones de implementación :

Beneficios Limitaciones
Actores Siempre actualizados con la Organización Mapeo estático que no incluye los datos de negocio/proceso
Filtros de Actor Mapeo dinámico que puede incluir los datos de negocio/proceso Se ejecutan sólo una vez al inicio de la instancia de la tarea
Los actores permiten actualizar automáticamente la visibilidad de las tareas cuando la organización cambia.
Los actores no pueden usar variables de proceso mientras que los filtros sí que pueden.
Los filtros se ejecutan sólo una vez cuando la tarea comienza.

Conectores

Permite a los procesos interactuar con otros sistemas de información externos (lo contrario que las APIs)

El Studio dispone de un conjunto de conectores

Los conectores personalizados pueden ser:

  • Implementados tanto desde el Studio como en otros Java IDEs
  • Importados desde el sitio web de la Comunidad
Los conectores se pueden poner a la entrada o a la salida de un proceso o de una tarea.
Cualquier tipo de dato o script de groovy puede ser utilizado como Entrada y salida.
Cuando es posible, se prefiere usar conectores antes que scripts para mejorar el rendimiento, manejo de errores (SP).
