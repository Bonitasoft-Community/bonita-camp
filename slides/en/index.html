Collaborate during the definition and execution of Business Processes

Bonita BPM Studio : Diagram Editor

Studio is an Eclipse based IDE. Requires a JVM 7 (not compatible with 8).
Allows creation of process diagrams which contain one or several processes.
Different steps: process modelling, data setup, form design, connector setup...
Diagram editor perspective: white board on which you drag and drop BPMN elements from a palette.

Bonita BPM Studio : Form Editor

Form editor perspective: form grid on which you drag and drop widgets from a palette

Bonita BPM Engine

In charge process execution

Based on Java 7 related technologies :

  • JDBC (via Hibernate and data sources)
  • Java EE 6 : JSP, Servlet, JTA ...
  • Spring
  • Quartz

Packaged with the Bonita Portal in a Java web application

Requires :

  • A Java application server
  • A JDBC compliant transactional database
No built-in UI
All interactions with the Engine from other components (Portal or other) are managed with Bonita APIs.

Bonita BPM Portal

End user and administration web interface

Displays the list of available tasks

Allows to deploy and configure processes

Users: start new cases, execute tasks
Administrators: deploy and configure process, manage access rights, track process execution
Portal uses the Bonita APIs as any client of the Engine

BPMN Introduction

Standard notation (OMG) for process modelling

A process is represented by a Pool

Key elements :

  • Start Event
  • Human Task
  • Automated Task
  • End Event
OMG = Object Management Group, founded several standards such as UML
Shapes (BPMN standard): circle = event, rectangle = activity
Colors (Bonita specific): green = start, blue = intermediate, red = end

Data Management : Scope

3 encapsulated levels of visibilities for data:

  • Process
  • Step
  • Form
- Data from upper scopes can be accessed (reverse is not possible).
- Data is always attached to a single process instance.
- Data life cycle is linked to its scope: once the scope is left (process, step...), data can no longer be modified.
- If business data must be shared between several processes: use an external DB or BDM (SP).

Data Management : Persistence and Types

All data is stored in a database by the Engine except for transient step data and form data

Any Java Serializable type can be used

Types such as primitives and Strings are serialized in order to be persisted

DB storage guarantees that data is safely kept in case of server shut down (expected or not).
Never try to R/W data directly from Bonita DB, use the APIs to do so.

BPMN Introduction (Continued)

3 types of gateways :

  • Exclusive
  • Parallel
  • Inclusive

Best practice: gateways should be placed in a symmetrical manner when possible

- Exclusive: use default TR and make sure only one condition is true among outgoing TRs
- Exclusive: only one incoming TR should reach the gateway.
- Parallel: all outgoing TR fired at the same time (no condition allowed).
- Parallel: all incoming TR are waited for even if some can never be reached due to bad design.
- Inclusive: used when dealing with parallel gateways with conditions

Forms

2 types of forms:

  • Case start form and step forms
  • Overview form

A form is a set of widgets placed on a grid

Case start and overview form are defined at pool level.
Click on submit button in forms triggers the completion of human tasks or the start of a process.
Overview form is always available: whether the process is running or completed.
Multiple page forms: pageflows

Widgets

Widgets carry a set of configuration options :

  • Name (id) and label
  • Hint, tooltip, example
  • Initial values, available values
  • Output operation
  • Conditional insertion, mandatory, read only...
  • Data validation

Actors

2 implementation options :

  • Actors : mapping between the process definition and the organization
  • Actor filters : logic executed at task start-up
Actors allow to automatically update task visibility when organization changes.
Actors cannot use process data whereas filters can.
Filters are executed only once when task starts.

Connectors

Allows processes to interact with the IS (opposite of the APIs that allow to pilot the Engine)

A set of standard connectors is shipped in the Studio

Custom connectors can be implemented using the Studio or other Java IDEs

A connector is composed of:

  • a definition (Java interface) which specifies inputs/outputs
  • an implementation (Java class) which defines the business logic
Connectors can be placed at the start or the end of a process or a task.
Any data type or groovy script can be used as I/O.
When possible prefer connectors over scripts for increased performance, error handling (SP).
