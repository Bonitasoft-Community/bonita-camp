Gateways

3 types of gateways:

Exclusive
Parallel
Inclusive

Conditions are defined on gateway outgoing transitions. No condition for parallel gateways

Gateways should be placed in a symmetrical manner when possible:

10

Exclusive: use default TR and make sure only one condition is true among outgoing TRs
Exclusive: only one incoming TR should reach the gateway.
Parallel: all outgoing TR fired at the same time (no condition allowed).
Parallel: all incoming TR are waited for even if some can never be reached due to bad design.
Inclusive: used when dealing with parallel gateways with conditions
Best practice: explicit, symmetric and default

Architecture overview

Sending data from the form to the business variable

2.30

When the user inputs some data in a form they are first stored in JavaScript variables.
When user click on the submit button, a REST API call is performed to the Engine API to instruct the Engine to start a new process instance or execute a task. This call carry the value of a JavaScript variable associated with the submit button. REST API call is an HTTP request that carry out data as JSON.
Engine will check that received JSON data match the contract definition and validation rules. If it match, the data are now stored in the contract variables.
Contract variables can then be used in various place such as a business variable default value (this will create a new row in business database), operation on a task to change the attribute value of a business variable (this will perform an UPDATE SQL request on the business database)...

Architecture overview

Display process variable in a form

8

The UI usually displays business information that is stored
[Studio & UID & H2 interface - explained the following based on exercise solution, run it with DEV TOOL enabled]
When you create a new form you can use some predefined JavaScript variables to make the link with business variables and documents linked to the context
The "context" variable give some information about the task and process instance associated with the visible form.
"context" variable have attribute that match the name of process variable with suffix _ref. Those variable have a "link" attribute that define the URL of the API to call in order to retrieve the business variable value.
The business variable value is actually a reference to a row in a table of the business database.
So to retrieve the business variable in a form, create a variable with type set to "External API" and the URL to ../{{context.myBusinessVar_ref.link}}. You can then map this variable to the form widgets.

Actors

Declared at pool level. Associated with lanes or human tasks. 2 options available: actors and actors filters


Benefits Limitations
Actors Always up to date with organization Mapping that can not rely on business/process data
Actor filters Mapping that can be based on business/process data Executed only once at task instantiation
12

Explain what is a candidate and assignee
Actor can be related to process instantiation, single human task or lane
[Studio - actors list and reference to process, initiator]
Actors allow to automatically update task visibility when organization changes.
Actor can be mapped to a subset of the organization using actor mapping: lists of users, groups, roles and memberships
[Studio & Portal - edit process configuration actor mappings, run it]
Actors cannot use any live data (business variables, external system information...) whereas filters can.
Filters are executed only once when task starts. They can be run again with API call.
[Studio & Portal - change task actor to actor filter, run it]
If you already have information about your organization stored in an LDAP/Active Directory server Subscription edition provide a tool that will save you time by dealing with the synchronization between LDAP/AD and Bonita user database.
On the authentication purpose, Bonita provides features to use SSO (SAML V2, CAS) and LDAP. Kerberos is also something we see regularly implemented for projects.
